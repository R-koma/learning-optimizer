# ADR-0001: フロントエンドフレームワークにNext.js + TypeScriptを採用する

## Status

Accepted

## Context

Learning Optimizer（LLMを活用した対話型学習支援システム）の本実装にあたり、フロントエンドフレームワークを選定する必要がある。

### 背景

- PoCではReact 19 + Vite 7でSPAとして実装し、WebSocketによるテキスト/音声チャットの双方向通信が動作することを検証済み
- 本実装では、PoCにはなかった以下の要件が加わる：
  - **認証機能**: ユーザーごとの復習スケジュール永続化のためにログイン機能が必要
  - **BFF（Backend for Frontend）層**: OAuth処理をフロントエンド側で完結させ、バックエンドにはJWTによるセッショントークン検証のみを担わせるクリーンなアーキテクチャを実現したい
  - **復習スケジュール管理UI**: エビングハウスの忘却曲線に基づいた復習タスクの表示・管理

### 判断軸（優先順位順）

1. **リアルタイム通信との相性**: コア機能がWebSocketベースのチャット（テキスト＋音声）であり、これが最優先
2. **将来の拡張性**: 認証統合、状態管理の複雑化、BFF層の構築に対応できるか
3. **開発速度**: 本実装フェーズであり、PoCからの移行コストも考慮
4. **キャリア投資価値**: 優先度は最低だが、業界標準の技術スタックであるかは考慮に値する

## Decision

**Next.js（App Router）+ TypeScript を採用する。**

PoCで使用していたReact + Vite（SPA）からNext.jsに移行し、言語はJavaScriptではなくTypeScriptを使用する。

## Reasons

### 検討した選択肢

#### Option A: React + Vite（SPA） — PoCの継続

PoCで検証済みのスタックをそのまま継続する案。

**メリット:**

- 移行コストゼロ。PoCのコードをそのまま発展させられる
- Viteの高速なHMRによる開発体験の良さは実証済み
- SPAとしてWebSocket通信との相性に問題がないことも確認済み

**デメリット:**

- 認証の実装を自前で構築する必要がある。react-router + 認証ライブラリ（Clerk, Auth0等）の組み合わせでOAuthフローを自前管理することになり、セッション管理やトークンリフレッシュのボイラープレートが増加する
- BFF層を持たないため、OAuthのclient secretをフロントエンドで安全に扱えない。結果としてバックエンド側にOAuthエンドポイントを実装するか、サードパーティの認証サービスに完全に依存する構成になる
- ルーティングやレイアウト管理がアプリの成長に伴い複雑化する可能性がある

#### Option B: Next.js（App Router） ✅ 採用

React Server Components（RSC）、Server Actions、App Routerなど最新のReact機能を活用できるフルスタックフレームワーク。

**メリット:**

- **認証統合**: Better AuthによりOAuthプロバイダー統合が宣言的に実装でき、セッション管理やトークンリフレッシュがフレームワークレベルで処理される。サーバーサイドでclient secretを安全に管理でき、フロントエンドにはセッショントークンのみを返す設計が自然に実現できる
- **BFF層**: API Routes（Route Handlers）により、フロントエンドと同一プロセス内にサーバーサイドのエンドポイントを構築できる。OAuthコールバック処理やトークン変換をNext.js側で完結させ、バックエンドにはJWT検証のみを担わせるクリーンな責務分離が可能
- **ファイルベースルーティング**: `app/` ディレクトリの規約により、ページ構成・レイアウト・ローディング状態が直感的に整理される。学習画面、復習ダッシュボード、ノート一覧といった画面の追加が構造的に容易
- **React Server Components**: 復習スケジュール一覧やノート表示など、初期描画をサーバーサイドで行い、インタラクティブな部分（チャットUI）のみクライアントコンポーネントにする最適化が可能

**デメリット:**

- PoCからの移行コストが発生する。ただし、PoCのReactコンポーネント（チャットUI等）は`"use client"`ディレクティブを付与すればほぼそのまま再利用可能であり、移行コストは限定的
- Next.jsのビルド・デプロイ構成がVercel以外では追加の設定を要する場合がある
- App Routerのサーバー/クライアントコンポーネントの境界管理は学習コストを伴う
- WebSocketはNext.jsのサーバー機能ではネイティブにサポートされないため、バックエンド側への直接接続を維持する構成になる。これはデメリットではなく設計上の制約として認識する

### 選定理由の要約

Next.js（App Router）を選定した主な理由は以下の通り:

1. **認証アーキテクチャの実現**: 本プロジェクトで目指す「OAuthをNext.jsで完結 → バックエンドはJWT検証のみ」という構成が、Better Auth + API Routesにより最も自然に実現できる。React + Vite（SPA）ではこの構成を実現するために大幅な追加実装が必要になる
2. **リアルタイム通信への影響が限定的**: WebSocket通信はバックエンド側への直接接続を維持するため、フレームワーク変更による影響はない。チャットコンポーネントは`"use client"`としてPoCのコードをほぼそのまま移植可能
3. **拡張性**: ファイルベースルーティング、レイアウトシステム、RSCにより、復習ダッシュボードやノート管理といった今後の画面追加に対して構造的にスケールする

### 言語選定: TypeScript vs JavaScript

#### Option A: JavaScript

**メリット:**

- 型定義の記述が不要なため、初期の記述量が少ない
- ビルド設定がシンプル（トランスパイルの考慮が不要）

**デメリット:**

- WebSocketメッセージの型安全性が保証されない。本アプリではバックエンド側との間で複数のメッセージ型（テキストチャット、音声データ、ノート生成結果、フィードバック等）をやり取りするが、これらの構造をコードレベルで保証する手段がない。ランタイムエラーとして初めて問題が発覚する
- リファクタリング時に変更の影響範囲を静的に把握できない

#### Option B: TypeScript ✅ 採用

**メリット:**

- **WebSocketメッセージの型安全性**: バックエンド側とやり取りするメッセージ（チャット応答、ノート構造、フィードバック等）をインターフェースとして定義することで、フロントエンド側のパース処理やUI表示のミスをコンパイル時に検出できる。バックエンド（Pydantic）のスキーマとフロントエンドの型定義を対応させることで、API契約が明示的になる
- **状態管理の信頼性**: 学習フローの各フェーズ（`idle`, `learning_dialogue`, `generate_note`等）をユニオン型やdiscriminated unionで表現でき、UI側でフェーズごとの表示切り替えを網羅的かつ安全に実装できる
- **Better Authとの統合**: Better AuthはTypeScriptをファーストクラスでサポートしており、セッション型の拡張（カスタムフィールドの追加等）が型安全に行える
- **Next.js App Routerとの親和性**: Next.jsの`create-next-app`はデフォルトでTypeScriptを推奨しており、App RouterのAPI（`generateMetadata`, `generateStaticParams`等）の型定義が公式に提供されている。TypeScriptで開発することがNext.jsの想定するファーストパスである
- **リファクタリング耐性**: 本実装フェーズではPoCのコードを再構成しながら機能追加を行うため、型による変更影響の静的検出が開発速度に直結する

**デメリット:**

- 型定義の初期記述コストが発生する。ただし、本アプリの規模では型定義が複雑化するほどのドメインモデルは限定的であり、コストは許容範囲内
- PoCがJavaScriptで書かれているため、移行時に型付けの作業が追加で発生する

#### 言語選定の理由の要約

TypeScriptを選定した主な理由は、本アプリの中核であるWebSocket通信のメッセージ型安全性と、LangGraphのフロー状態に対応するUI状態管理の信頼性を確保するためである。加えて、Next.js（App Router）およびBetter AuthがTypeScriptをファーストクラスでサポートしていることから、フレームワーク選定との一貫性が保たれる。

## Consequences

### 良い影響

- Better Authによる認証実装の簡素化と、OAuth処理のサーバーサイド完結
- API Routesを活用したBFF層により、フロントエンド ↔ バックエンド間の責務が明確に分離される
- ファイルベースルーティングにより、画面の追加・整理が規約に沿って行える
- React Server Componentsにより、チャット以外の画面（ノート一覧、復習スケジュール等）のパフォーマンス最適化が可能
- TypeScriptにより、WebSocketメッセージやフロー状態の型安全性が担保され、バックエンド（Pydantic）とのスキーマ整合性が明示的に管理できる
