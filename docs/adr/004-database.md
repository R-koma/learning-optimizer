# ADR-0003: データベースおよびデータアクセス方式にPostgreSQL/SQLファーストアプローチを採用

## Status

Accepted

## Context

Learning Optimizer（LLMを活用した対話型学習支援システム）の本実装にあたり、データベースおよびデータアクセス方式を選定する必要がある。

### 背景

- PoCではノートデータとセッション状態がインメモリ（プロセス内辞書）で管理されており、サーバー再起動で全データが消失する課題がREADMEに記載されている
- 本実装では永続化が必須であり、以下のデータを保存する必要がある：
  - **ユーザーデータ**: BetterAuth経由の認証情報（ADR-0001, ADR-0002で決定済み）
  - **ノート**: 対話セッションから自動生成される構造化データ（Structured Output）
  - **フィードバック**: 専門家LLMによる理解度評価（high/medium/low）、良かった点、改善点
  - **復習スケジュール**: エビングハウスの忘却曲線に基づくスケジューリングデータ
  - **LangGraphチェックポイント**: 学習フローの状態（phase、対話履歴等）の永続化。PoCではインメモリだったため、WebSocket切断やサーバー再起動でフロー状態が消失していた
- ADR-0002にて、バックエンドにFastAPI（Python）モノリスを採用する方針が決定済み

### 判断軸（優先順位順）

1. **LangGraphのチェックポイント機能との親和性**: LangGraphの公式チェックポイントライブラリがサポートするDBであること
2. **ノートのセマンティック検索**: 将来的にノート間の類似検索やトピック横断の知識グラフ構築に対応できるベクトル検索機能
3. **運用のシンプルさ**: マネージドサービスの利用可否、バックアップ・スケーリングの容易さ
4. **学習コスト・既存経験**: 自身のスキル習得も含めた判断

---

## Decision

### 1. データベース

**PostgreSQLを採用する。** pgvector拡張によるベクトル検索も将来的に活用する。

### 2. データアクセス方式

**SQLファーストアプローチを採用する。** ORMは使用せず、生SQLとマイグレーションツールでデータアクセスを行う。

---

## Reasons

### データベース選定

#### Option A: PostgreSQL ✅ 採用

**メリット:**

- **LangGraphチェックポイントの公式サポート**: `langgraph-checkpoint-postgres`が公式に提供されており、LangGraphのフロー状態（phase、対話履歴、中間結果等）をPostgreSQLに永続化できる。追加実装なしでチェックポイント機能を利用可能
- **pgvectorによるセマンティック検索**: pgvector拡張を導入することで、ノートのembeddingベクトルを同一DB内に保存し、類似ノート検索やトピック横断の知識発見が実現可能。専用のベクトルDBを別途運用する必要がない
- **リレーショナルデータとの統一管理**: ユーザー、ノート、フィードバック、復習スケジュールは全てリレーショナルな構造を持つ。これらとベクトルデータ、LangGraphチェックポイントを単一のPostgreSQLインスタンスで統一管理でき、運用の複雑さを最小限に抑えられる
- **BetterAuthの公式サポート**: BetterAuthはPostgreSQLへの直接接続をサポートしており、ユーザー・セッション・アカウントデータの保存先として自然に統合できる
- **マネージドサービスの豊富さ**: Supabase、Neon、AWS RDS、Cloud SQL等、PostgreSQLのマネージドサービスが多数存在し、運用負荷を軽減できる

**デメリット:**

- SQLiteと比較して、ローカル開発環境のセットアップにDocker等の追加手順が必要
- 個人プロジェクトの規模に対してはオーバースペックである可能性がある

#### Option B: MySQL

**メリット:**

- 世界で最も広く利用されているRDBMSの一つであり、ドキュメント、チュートリアル、コミュニティのナレッジが豊富
- 読み取り性能が高く、レプリケーションの設定がシンプル
- PlanetScale、AWS RDS、Cloud SQL等のマネージドサービスが利用可能
- BetterAuthがMySQLへの直接接続をサポートしている

**デメリット:**

- **LangGraphチェックポイントの公式サポートが存在しない**: LangGraphの公式チェックポイントライブラリはPostgreSQLとSQLiteのみをサポートしており、MySQL用のアダプターは提供されていない。チェックポイント機能を利用するにはカスタム実装が必要となり、開発コストとメンテナンス負荷が大幅に増加する
- **ベクトル検索の未成熟**: MySQLにはpgvector相当のネイティブなベクトル検索拡張が存在しない。将来的なセマンティック検索の導入に際して、専用のベクトルDB（Pinecone、Weaviate等）を別途運用する必要が生じる
- PostgreSQLと比較して、`JSONB`型、ウィンドウ関数、CTEなどの高度なSQL機能のサポートが限定的（MySQLも改善が進んでいるが、PostgreSQLが先行）

#### Option C: SQLite

**メリット:**

- セットアップ不要でファイルベースで動作し、開発環境の構築が最もシンプル
- `langgraph-checkpoint-sqlite`が公式に提供されている
- 個人利用のプロジェクトではパフォーマンスも十分

**デメリット:**

- ベクトル検索の拡張がPostgreSQLほど成熟していない。将来的なセマンティック検索の導入に制約が生じる
- 同時書き込みに制約があり、WebSocket経由の複数セッション同時処理でロック競合が発生する可能性がある
- マネージドサービスが限定的であり、本番デプロイ時のバックアップ・スケーリング戦略が別途必要

#### Option D: MongoDB

**メリット:**

- スキーマレスな設計により、LLMの構造化出力（ノート、フィードバック）の保存が柔軟
- 対話履歴のようなドキュメント型データとの親和性が高い

**デメリット:**

- LangGraphのチェックポイントライブラリが公式ではなくコミュニティ提供であり、サポートの継続性・安定性に不確実性がある
- リレーショナルなデータ（ユーザー ↔ ノート ↔ 復習スケジュールの関連）を扱う際にJOINが使えず、アプリケーション層での結合処理が必要になる

#### DB選定の理由の要約

PostgreSQLを選定した決定的な理由は、LangGraphチェックポイントの公式サポートとpgvectorによるセマンティック検索の両方を単一DBで実現できる唯一の選択肢であることだ。MySQLは広く利用されているRDBMSだが、LangGraphのチェックポイント公式サポートが存在しない点が本プロジェクトの最優先判断軸を満たさない。SQLiteはチェックポイントの公式サポートがあるものの、ベクトル検索と同時書き込みに制約がある。加えて、保存対象のデータが全てリレーショナルな構造を持ち、BetterAuthもPostgreSQLへの直接接続をサポートしているため、システム全体のデータ永続化をPostgreSQLに統一できる。

---

### データアクセス方式の選定

#### Option A: ORM（SQLAlchemy等）

FastAPIエコシステムではSQLAlchemyの利用率が高く、業界標準のデータアクセス方式として広く採用されている。

**メリット:**

- Pythonオブジェクトとテーブルのマッピングが自動化され、CRUD操作の記述量が減る
- マイグレーション（Alembic）との統合がスムーズ
- DB方言の差異を吸収し、将来的なDB変更に対する柔軟性がある
- FastAPIの公式ドキュメントやチュートリアルの多くがSQLAlchemyを前提としており、情報量が豊富

**デメリット:**

- ORMが生成するSQLがブラックボックスになりがちで、パフォーマンスチューニングやデバッグが困難になる場合がある
- pgvector等の拡張機能やPostgreSQL固有の機能をORMで扱う際に、追加の設定やワークアラウンドが必要になることがある
- `langgraph-checkpoint-postgres`はSQLAlchemy非依存で動作するため、ORMを導入するとDB接続管理が二重化する（SQLAlchemyのSessionとlanggraphのコネクションプール）
- BetterAuthも独自にDBスキーマを管理するため、ORM層のスキーマ管理とBetterAuthのスキーマ管理が衝突する可能性がある
- SQLそのものの理解が深まらない

#### Option B: SQLファーストアプローチ（生SQL + マイグレーションツール） ✅ 採用

**メリット:**

- **技術スタック全体との干渉の少なさ**: 本プロジェクトではLangGraphチェックポイント（`langgraph-checkpoint-postgres`）とBetterAuthがそれぞれ独自にDBアクセス・スキーマ管理を行う。ORMを導入すると、SQLAlchemyのSession管理・コネクションプール・スキーマ定義がこれらと競合するリスクがある。SQLファーストであれば、アプリケーション層のDB接続がシンプルなコネクションプール（`asyncpg`等）に統一され、LangGraphやBetterAuthとの共存が自然になる
- **SQL学習の機会**: 生SQLを書くことでSQLの理解が直接深まる。リレーショナルDB設計、クエリ最適化、インデックス戦略といった基礎スキルの習得に直結する
- **透明性**: 実行されるSQLが常に明示的であり、パフォーマンスの把握やデバッグが容易
- **PostgreSQL固有機能の活用**: pgvectorのベクトル操作、`JSONB`型、ウィンドウ関数等、PostgreSQL固有の機能を制約なく利用できる

**デメリット:**

- CRUD操作のボイラープレートコードが増加する。特にINSERT/UPDATE文の手書きは冗長になりやすい
- マイグレーション管理に別途ツール（Alembic単体、dbmate、golang-migrate等）の選定と設定が必要
- SQLインジェクション対策をパラメータバインディングで自前で担保する必要がある（ORMでは自動的に処理される）
- FastAPIエコシステムの主流（SQLAlchemy）から外れるため、参考にできる情報やチュートリアルが相対的に少ない

#### データアクセス方式の選定理由の要約

SQLファーストアプローチを選定した理由は2つある。第一に、本プロジェクトの技術スタック構成上の理由である。LangGraphチェックポイントとBetterAuthがそれぞれ独自にDB接続・スキーマ管理を行う環境において、ORMを追加導入するとDB接続管理の三重化（SQLAlchemy + langgraph + BetterAuth）が発生し、複雑さが増す。SQLファーストであればアプリケーション層のDB接続がシンプルに保たれる。第二に、SQL自体の学習機会の確保である。ORMの抽象化層を挟まないことで、実行されるSQLの透明性が保たれ、RDB設計・クエリ最適化のスキルが直接的に向上する。

#### 補足: 「一般的にはORMが推奨される」という反論への回答

一般的なWebアプリケーション開発において、SQLAlchemy等のORMをメインに据え、複雑なクエリにのみ`text()`で生SQLを書く「いいとこ取り」が現代のベストプラクティスとされている。その主な論拠は、大量のCRUD操作の保守性、SQLインジェクション防止の標準化、チーム開発時のスキル差の吸収である。

この一般論は正しいが、本プロジェクトには以下の固有事情があり、ORMの利益がコストを上回らないと判断した。

**1. DB接続の多重管理問題:**
一般的なWebアプリでは「アプリケーション → ORM → DB」の一本道だが、本プロジェクトではBetterAuth（独自の接続管理・スキーマ管理）とlanggraph-checkpoint-postgres（独自のコネクションプール）が既にPostgreSQLに対して独立したDB接続を管理している。ここにSQLAlchemyのSession管理・コネクションプールを追加すると、3つの独立したDB接続管理が同一DBに対して動作することになり、コネクションプールの枯渇やトランザクション管理の複雑化を招くリスクがある。SQLファーストで`asyncpg`を直接使用すれば、BetterAuthやlanggraphと同レベルの軽量な接続管理に統一でき、コネクションプールの共有・統一的な管理が容易になる。

**2. CRUDの規模が限定的:**
本アプリでアプリケーション層が実際にCRUDを書くテーブルはnotes、feedbacks、review_schedulesの3テーブル程度である（users/sessions/accountsはBetterAuthが管理、checkpointsはlanggraphが管理）。この規模ではORMによるボイラープレート削減効果は限定的であり、逆にORM導入・設定のオーバーヘッドの方が大きい。

**3. セキュリティ（SQLインジェクション防止）について:**
`asyncpg`のパラメータバインディング（`$1`, `$2`）を使用すれば、ORMと同等のSQLインジェクション防止が得られる。これはORM固有の利点ではなく、「文字列結合でSQLを構築しない」という規律の問題であり、SQLファーストでも十分に対応可能である。

## Consequences

### 良い影響

- LangGraphのチェックポイント、ユーザーデータ（BetterAuth）、アプリケーションデータ（ノート、フィードバック、復習スケジュール）を単一のPostgreSQLインスタンスに統一でき、運用がシンプルになる
- pgvectorにより、将来的なセマンティック検索（類似ノート検索、トピック横断の知識発見）への拡張パスが確保される
- SQLファーストにより、LangGraphチェックポイント・BetterAuth・アプリケーション層のDB接続が干渉なく共存できる
- 生SQLを書くことで、SQLのスキルが直接的に向上する。これはDB設計やクエリ最適化の知識として、他のプロジェクトにも転用可能

### 注意すべき影響

- マイグレーションツールの選定が別途必要。軽量なツール（dbmate等）の導入を検討する
- SQLインジェクション対策として、全てのクエリでパラメータバインディングを徹底する必要がある。コードレビュー時のチェックポイントとして意識する
- pgvectorの導入タイミングは、ノートのセマンティック検索が実際に必要になった段階で行う。初期リリースではリレーショナルデータの永続化を優先する
