# ADR-005: ノート手動編集と復習再生成の競合をレイヤ分離（追記専用）で解決する

## Status

Proposed

## Context

ノート編集機能（#134）の導入により、ユーザーが AI 生成ノートの `topic` / `summary` / `content` を手動編集できるようになる。一方、復習セッションは `graph/nodes/update_note_and_feedback.py` の `update_note_and_feedback` ノードで既存ノートを再生成する。この2つが同じ `notes.content` を書き換えるため、競合の設計判断が必要になった。

### 現状の再生成挙動（正確な把握）

`update_note_and_feedback` は次の手順で動く:

1. DB から既存ノートの `content` / `summary` を取得
2. それらを `UPDATE_NOTE_PROMPT` に「既存ノート」として注入（プロンプトには「既存内容を保持」「無理に更新しない」ルールあり）
3. LLM が改訂版を生成し、`note_repository.update` で `content` / `summary` を**置き換える**

つまり手動編集は「破棄」されるのではなく「LLM の入力に混ぜられ、LLM の裁量で書き直される」。ブラインドな上書きではない。

### 問題

手動編集テキストが AI 生成テキストと**同格**に扱われ、来歴（どちらが書いたか）を持たない。そのため LLM の言い換え・再構成・要約の過程で原文が失われたり削られたりしうる。1回の復習では残っても、復習を重ねるうちに手動編集が徐々に削られる**侵食（erosion）**が起きうる。ユーザーが手で直したというコストが、自動再生成によって無下にされるのは設計として望ましくない。

### 根本原因

> 1つのフィールド（`notes.content`）を、人間と AI という2人の writer が破壊的に共有しており、かつ来歴を持たない。

これは協調編集・AI 支援編集で典型的な問題であり、解は「来歴を持つ」＋「2人の writer が同じ領域を破壊的に書かない」に集約される。

### 判断軸

1. **手動編集の保全強度**: ユーザーの編集が確実に守られるか（プロンプト遵守頼みか、コードで保証されるか）
2. **改善の恩恵の維持**: 復習対話による内容改善を引き続き取り込めるか
3. **自動更新 UX の維持**: 「復習したらノートが更新される」現行の自動フローを壊さないか
4. **実装量と後方互換**: 既存の復習フロー・スキーマへの影響の小ささ

---

## Decision

**来歴フラグを土台に、ノート本文を「人間が所有する base」と「AI が書く復習アデンダ」にレイヤ分離し、追記専用（append-only）で扱う。**

- `notes` に来歴フラグ `manually_edited_at TIMESTAMPTZ NULL` を追加する（#134）。フラグはユーザー手動編集パスでのみセットし、復習再生成パスでは立てない。
- `manually_edited_at` が立っているノートの復習では、`content` 本体を書き換えず、AI 改訂は**追記分のみ**を生成する（#235）。
- **保全はプロンプト遵守ではなくコードで担保する**: 人間の base を LLM の出力に通さず、`base + addendum` をコードで連結することで base を byte 単位で保全する。
- `manually_edited_at` が無いノートは従来どおりフル改訂し、後方互換を維持する。

追記の格納方式は #235 着手時に次の2案から選定する（本 ADR では未確定）:

- **案 i**: `content` 末尾の専用セクション（例 `## 復習で深まった点`）にコードで追記する
- **案 ii**: append-only な別テーブル `note_revisions` に復習ごとの改訂を保存し、表示時に base + 改訂履歴を合成する

実装は2段階に分ける:

- **#134**: 編集 UI ＋ 来歴フラグの追加・セット（土台）
- **#235**: フラグを用いた base 保全・追記専用の再生成ロジック（本体）

#235 は #134 の来歴フラグに依存するため、#134 を先に実装する。

## Reasons

### 検討した選択肢

すべての案に共通する土台は来歴フラグである。来歴が無ければ「どのノートを保護すべきか」を判断できない。

#### Option A: ロック（手動編集ノートは復習で content 再生成をスキップ）

手動編集されたノートは、復習で `content` を再生成せずスキップする（フィードバック・スケジュールは更新）。

**メリット:**

- 実装が最小。侵食を確実に止められる
- フラグは A/B 共通の土台のため無駄にならない

**デメリット:**

- 一度でも編集すると、そのノートは以後ずっと復習による改善の恩恵を受けない（all-or-nothing の凍結）
- 判断軸2（改善の恩恵の維持）を満たさない

対応するモダンパターン: Terraform の `lifecycle { ignore_changes }`、セルのピン留め。

#### Option B: レイヤ分離（追記専用） ✅ 採用

人間が所有する base をコードで保全し、AI は復習アデンダにのみ追記する。

**メリット:**

- 手動編集を**コードレベルで**保全できる（人間部分を LLM 出力に通さないため byte 単位で保全＝侵食ゼロ）。プロンプト遵守頼みではない
- 復習による改善の恩恵を捨てない（AI は追記を続けられる）。A の凍結弱点を回避
- 「復習でノートがどう育ったか」の履歴が可視化でき、プロテジェ効果アプリとして学習の成長が見えるプロダクト価値が出る（特に案 ii）
- C ほどの UX 変更・実装量を要さず、自動更新フローを壊さない

**デメリット:**

- スキーマ・プロンプト・描画・復習ノードの変更が必要（A より実装量が多い）
- base と addendum の表示構成を決める必要がある

対応するモダンパターン: イベントソーシング、単一 writer-per-region（CRDT 的発想）、Notion AI の追記。

#### Option C: サジェスト（差分承認）

復習は上書きせず改訂案を別保存し、ユーザーが差分を見て採用 / 却下 / 手直しする。

**メリット:**

- 最強の保護。ユーザーの明示的合意なしに何も変わらない
- 信頼性の高い UX

**デメリット:**

- 実装量が最大（保留改訂の保存、差分 UI、承認 API）
- 「復習したらノートが更新される」自動 UX が「要承認」に変わり、現行フローを壊す
- 判断軸3（自動更新 UX の維持）を満たさない

対応するモダンパターン: PR レビュー、Google Docs の suggesting モード、git の 3-way merge。

#### Option E: 完全自動の 3-way マージ（不採用）

最後の AI baseline・現在のノート・新 AI 改訂を git 風に自動マージする。

散文（Markdown 本文）に対する自動 3-way マージは境界が曖昧でマージ品質が安定しない。テキスト diff を当てるより、領域を分けて単一 writer にするほうが堅牢なため採用しない。

### 採用理由の要約

判断軸1（保全強度）と判断軸2（改善の恩恵）を同時に満たすのは B のみである。A は侵食を止めるがノートを凍結し改善の恩恵を失う。C は保護も UX も優れるが実装量が大きく、自動更新フローを壊す。B は「人間が編集した領域は AI が触れず、AI は別領域に追記する」という単一 writer-per-region の原則を、コードによる連結で堅牢に実現でき、自動更新 UX も維持する。加えて復習による成長履歴という学習アプリらしい価値も得られる。

---

## Consequences

### 良い影響

- ユーザーの手動編集コストがコードレベルで保護され、復習を重ねても侵食されない
- 復習による内容改善を引き続き取り込める（凍結しない）
- 未編集ノートは従来どおりフル改訂され、後方互換が保たれる
- 案 ii を採れば「復習でノートがどう育ったか」の履歴が残り、学習の成長が可視化される

### 注意すべき影響

- `note_repository.update` は復習ノードとユーザー PATCH の両方から呼ばれる。来歴フラグを立てるのはユーザー編集パスのみとし、復習パスで誤って立てないよう、ユーザー編集専用の更新経路を用意する必要がある
- 案 ii を採る場合、ノート詳細ページ（`client/app/(main)/notes/[id]/page.tsx`）の描画を base + 改訂履歴の合成表示に変更する必要がある
- `aspect_map`（観点マップ）は本文編集時に再生成されないため、編集後に古いまま残りうる（本 ADR の対象外の既知の制約）
- 追記格納方式（案 i / 案 ii）は #235 着手時に確定する。確定後、必要なら本 ADR を更新するか追補 ADR を残す

### 関連

- #134 ノート編集機能（編集 UI ＋ 来歴フラグの追加・セット）
- #235 復習再生成で手動編集ノートを侵食しない（レイヤ分離・追記専用）
